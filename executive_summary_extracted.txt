Executive Summary
The Agentic AI Video Call–Based Onboarding system is a complex ML-driven loan origination platform
that integrates real-time video KYC, consent capture, computer vision, speech-to-text (STT), and language
understanding to automate credit decisions. This report outlines a full Software Requirements Specification
(SRS) and development plan. We begin with purpose, scope and definitions, then detail stakeholders, overall
description, functional and non-functional requirements, data needs, constraints, and risk mitigations. We
present a high-level architecture with a mermaid diagram, including component interactions and data flow.
A recommended tech stack (Python/FastAPI, Next.js, PyTorch, etc.) is justified with citations. We
propose a clean project structure with CI/CD (using GitHub Actions or similar) and an automated testing
strategy (unit, integration, end-to-end). A detailed API specification lists REST endpoints, payload
schemas, and auth. The ML pipeline is outlined (data preprocessing, model choice, training, metrics,
serving, versioning) with best practices. We include deliverables, milestones and a Gantt chart
(mermaid) plus a table of alternative design choices (e.g. FastAPI vs Flask) with trade-offs and citations
. Finally, we provide an “Antigravity” script (a high-level bash/Python scaffold) to bootstrap the project
codebase and infrastructure, with instructions to run it. All claims and recommendations are supported by
authoritative sources.
1. Software Requirements Specification (SRS)
1.1 Purpose
The purpose of this SRS is to define all requirements for the Video-Based Digital Loan Origination & Risk
Assessment System. It will guide development by clearly stating what the system must do, ensuring
alignment among stakeholders (business, legal, devs, testers) and reducing ambiguity. Specifically,
the system must enable secure, compliant real-time customer onboarding via video, verify identity and
intent, evaluate risk, and generate personalized loan offers automatically. 
1.2 Scope
The scope covers a web/mobile application with a video-call interface for loan applicants, back-end AI/ML
services for KYC (face/ID verification, age estimation), STT and LLM-based analysis of verbal inputs, credit-
risk evaluation using internal and external data, and offer generation. It excludes unrelated banking
processes (e.g. loan servicing post-approval). This SRS defines both the functional requirements (what the
system does) and non-functional requirements (how well it performs, e.g. security, scalability).

1.5 Overall Description
1.5.1 User Needs
Users need a quick, seamless loan application process with instant decisions. They must trust the system to
be secure and fair . Business needs include fraud reduction, compliance, and automation to reduce costs. 
1.5.2 Assumptions and Dependencies
Assumptions: Users have webcam/microphone and connectivity. External services (credit bureau
APIs, identity databases, speech recognition, LLM APIs) are available. User privacy regulations
(GDPR/KYC) will be followed. 
Dependencies: Cloud infrastructure (e.g. AWS/GCP), third-party APIs (Twilio/Jitsi for video, Google/
Meta ML services for STT/LLM if used), internet. 
1.5.3 System Context
The system interfaces with external credit bureaus and policy databases, and internally comprises front-end
(web/mobile), back-end (API server , ML services) and database layers. Data flows from user (video,
audio, metadata) through the back-end (CV analysis, STT, LLM, risk engine) to produce loan decisions.

2.1 Functional Requirements (Features)
2.1.1 User Onboarding Video Call (Use Case)
FR1: Real-Time Video Call: The system shall support a secure web-based video call between the user
and an agentic AI interface. (The front-end uses WebRTC or third-party video SDK.) 
FR2: KYC Verification: During the call, the system shall capture the user’s photo/ID and perform
face recognition against an official ID database to confirm identity (CV module). 
FR3: Age Verification: The system shall estimate the user’s age via facial analysis and verify it meets
policy (e.g. age ≥ 18). 
FR4: Verbal Consent Capture: The system shall clearly ask for explicit verbal consent for terms and
verify using STT that consent is given; log consent transcript. 
FR5: Data Extraction: The system shall parse user-provided data (spoken or visual) for loan-relevant
details: income, loan purpose, employment, etc. (Use NLP/LLM to extract structured data). 
FR6: Auto-Fill: The system shall auto-populate the loan application form with extracted data and
allow user confirmation. 
FR7: Risk Evaluation: The system shall retrieve external data (credit bureau scores, policy rules) and
compute an aggregate risk/propensity score for the user . 
FR8: LLM Insight: The system shall use an LLM (e.g. GPT) to interpret customer’s verbal responses
and classify them (e.g. risk band, confidence). 
FR9: Offer Generation: Based on policies, risk score, and user data, the system shall generate
personalized loan offers (amount, tenure, EMI) and present them to the user . 
Use Case – “Jane applies for a loan”: Jane visits the platform, starts a video call, consents verbally, and
answers questions. The system captures her ID (FR2), verifies age (FR3), extracts that she earns \$X (FR5),
computes her risk (FR7), and instantly offers a \$Y loan (FR9). 
2.2 External Interface Requirements
UI Interface (Human): The frontend (web or mobile) shall provide a video-call UI (integration via
WebRTC/Twilio). It shall display forms (auto-filled), consent dialogs, and final offers. The UI must be
accessible and responsive. 
API Interface (System): The back-end shall expose RESTful APIs (or GraphQL) for all front-end
interactions: user session management, data submission, retrieving results. 
Hardware Interfaces: Use of standard camera/microphone on user device. Back-end servers will
interface with GPUs (for ML inference) and with data storage (DB, S3). 
Third-party Integration: The system shall interface with: 
Identity Verification API (for KYC), 
Speech-to-Text engine (e.g. Whisper or cloud STT) for audio transcription, 
LLM service (OpenAI/GPT or local model) via API, 
Credit Bureau API for credit score and history. 
2.3 Non-Functional Requirements (Quality Attributes)
Performance: The system shall process video and audio in real-time (e.g. end-to-end onboarding < 5
minutes). API responses (e.g. offer generation) should return in <1 second. Throughput must handle
hundreds of concurrent users (scale horizontally).

Data Sources:

Logs: Decision logs and consent transcripts stored in an append-only log for audit (retain 5+ years

Policies/Config: Version-controlled (Git or DB) for reproducibility.

models. 
Budget: Third-party API costs (STT/LLM) and cloud resources must be justified.

2.9 Risk Analysis & Mitigation
Key risks include: 
Data Quality Risk: Poor video/audio quality may degrade model outputs. Mitigation: Implement
input checks (e.g. ensure face visibility), allow manual override; prompt user to retry. 
Model Inaccuracy: ML models may yield false positives/negatives (e.g. mis-identifying age or risk). 
Mitigation: Use ensemble of models, include rule-based fallbacks, threshold tuning and regular re-
training with new data. 
Security Breach: Personal data leakage. Mitigation: Encrypt all PII, use secure key management,
perform code reviews and pen tests.

The User interacts via a web front-end (Next.js/React). 
Video Call is managed by a service (e.g. WebRTC server or Twilio); streams go into the backend. 
The Backend (FastAPI) handles all logic: splitting video frames to the CV module (face recognition,
ID check) and Age Estimation, sending audio to STT (e.g. Whisper or Google STT) whose text goes
into an LLM. All results feed the Risk Engine and Offer Generator. 
A central Database stores user profiles, transcripts, logs, and offers; a Cache (Redis) holds session
data and speedup. 
External APIs (Credit Bureau, internal Policy DB) provide data into the risk calculation. 
Data Flow (Sequence)
Start Session: User logs in/initiates call – backend creates session token. 
Video/Audio Capture: User connects camera/mic. The frontend streams data to the Video Service
which relays to backend. 
KYC Checks: Backend captures video frames → CV module. It compares user’s face to ID database
(FR2) and logs result. Simultaneously, extracts document data (e.g. from an ID photo). 
Consent & Info Capture: The system prompts (via synthesized voice or on-screen) for consent and
key info. User speaks answers; audio sent to ASR module. 
Speech-to-Text & NLP: Audio → STT (fast, low latency) to get transcript. Transcript → LLM (fine-
tuned for this domain) to extract structured info (income, purpose, sentiment). LLM is
constrained by context to prevent hallucination. 
Risk Calculation: Backend calls Credit Bureau API (if allowed) for credit score, fetches policy rules.
The risk engine uses bureau data, LLM outputs (e.g. confidence level), and CV age-check to compute
a risk score. 
Offer Generation: Based on risk and policy, the system formulates loan terms. Results (approved
amount, rates) are stored in DB and sent to frontend. 
End Session: Backend logs consent and decision. User receives offers, may accept or request
changes. Final decision stored as audit record. 
The architecture and flow ensure modularity and reuse: e.g. CV, ASR, LLM can be independent microservices
(perhaps Docker containers). This decoupling aids testing and scaling.

4. Technology Stack Recommendations
We choose technologies that are production-proven, scalable, and well-supported: 
Frontend: Next.js (React) – modern framework for web apps. It supports server-side rendering
(SEO, faster first load) and integrates well with React components. It’s ideal for building responsive
video-call UIs. 
Backend: Python (FastAPI) – high-performance async framework. FastAPI can handle ~15k+ req/s
vs Flask’s ~2-3k, important for real-time throughput. It provides built-in data validation (via
Pydantic) and auto-generates OpenAPI docs, reducing boilerplate and errors. Developers can
leverage Python’s rich ML ecosystem. 
Machine Learning: 
Deep Learning Frameworks: PyTorch (or TensorFlow) for model development. PyTorch has strong
community support and dynamic graph for ease of experimentation. 
Computer Vision: OpenCV, dlib, or specialized libraries (e.g. FaceNet) for face recognition/age
estimation. Pre-trained models (e.g. OpenCV’s DNN modules) can accelerate dev. 
Speech-to-Text: Consider OpenAI Whisper (open-source), Google Cloud STT, or AWS Transcribe for
high-accuracy transcription (often using cloud APIs for better performance on noisy audio). 
LLM Integration: Use a high-quality language model via API (e.g. GPT-4 or Claude) or an open
checkpoint (via Hugging Face). The LLM will be used in a controlled way to parse spoken answers. 
Database: PostgreSQL (relational) for user/account data, because of ACID compliance for financial
data. Or a document DB (e.g. MongoDB) if flexibility needed for diverse unstructured logs. Redis for
caching sessions and intermediate results. 
APIs and Services: 
Video Call: Twilio Video or Jitsi for scalable video sessions (offloading WebRTC complexities). 
Identity Verification: If possible, tie into existing KYC providers (e.g. Jumio, Onfido). Otherwise use
CV models to match ID photos. 
Credit Bureau: Integration via REST API to Experian/TransUnion/Equifax. 
Deployment/Cloud: AWS or GCP. For example, AWS with Elastic Beanstalk or EKS for services, S3 for
storage, RDS for PostgreSQL, and Lambda/ECS for microservices. 
Authentication/Authorization: JWT tokens for API security, OAuth2 for user login (if applicable).
Use industry-standard libraries. 
Rationale: FastAPI’s async nature and auto-docs markedly improve development speed and performance
. Python/PyTorch are industry-standard for ML. Next.js offers a robust, SEO-friendly UI. A combination of
relational (for structured data) and NoSQL (for logs/media) plus caching covers all data needs. Cloud-
managed services ensure scalability.

7. ML Pipeline
The ML pipeline has two main flows: training/offline and inference/online. 
Data Preprocessing: Gather labeled data for training (e.g. customer videos with annotated
transcripts and known outcomes). Preprocess steps include video frame sampling, image
normalization, audio denoising, text cleaning. For tabular data (credit records), normalize and
encode. Tools: Python scripts (Pandas, OpenCV, librosa). Automate with Airflow or Prefect. 
Model Selection:
Face Recognition & Age Estimation: Use pre-trained CNNs (e.g. FaceNet for identity, DeepFace or
SSR-Net for age) and fine-tune on relevant demographic data. 
Speech-to-Text: Leverage a state-of-the-art ASR model (e.g. Whisper) for robust transcription. 
Intent/Info Extraction (LLM): Fine-tune or prompt an LLM (e.g. GPT) to parse audio transcripts into
structured info. This may use few-shot or domain adaptation. 
Risk Model: A classification/regression model (e.g. XGBoost, or NN) trained on historical loan data to
predict default probability. Input features: bureau scores, declared income, conversation features
(confidence signals from LLM), etc.

Training: Use train/validation/test splits. Employ k-fold or cross-validation to ensure generalization.
For small labeled data, consider data augmentation (simulate audio noise, faces). Use GPU/TPU
resources (e.g. AWS SageMaker). Log experiments with MLflow for tracking. 
Evaluation Metrics: Depends on task: 
CV tasks (face match): Accuracy, False Acceptance Rate/False Rejection Rate. 
Age estimation: Mean Absolute Error (years). 
STT: Word Error Rate (WER). 
LLM extraction: F1-score on extracted fields. 
Risk model: AUC-ROC, precision/recall, confusion matrix. 
Overall system: Offer acceptance rate, decision latency. We must also monitor bias/fairness metrics
(e.g. error rates across demographics). 
Model Serving: Deploy models as REST or gRPC services. For real-time inference, use lightweight
frameworks (e.g. TorchServe or FastAPI endpoints). Containerize each ML service. Ensure versioning:
each deployed model version is tagged (e.g. via MLflow or a model registry) and audit-tracked. Allow
rollback to previous model. 
Monitoring (ML-Ops): Track inference performance (latency) and drift: compare new data
distributions to training baselines. Set thresholds to trigger retraining. Maintain data provenance. A/
B testing can be used when releasing new models. 
In summary, the ML pipeline is orchestrated end-to-end (data to deployment) with validation at each step
